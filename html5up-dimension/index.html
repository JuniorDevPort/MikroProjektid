TDD 10.22

Let's talk about TDD. I think TDD is a complicated concept to grasp for a junior or a beginner in software development. It is a more advanced way of looking at problem-solving and has an engineering aspect to it. It is to make a better design. You write the test before you write the code.

To write code in a TDD way, you need to write the test before the code. You need to look at the specifications. The specifications of the functionalities of the end system the client is looking for. what it is that the software is supposed to deliver to the end-user. these specifications solve some kind of problem. the problem is why the software is being written. these problems that the end-user needs to be solved are sometimes quite complex so you need to break these problems down into smaller ones.

For simpler problems, you can perhaps write a function to solve. you need to break down the problems more and more until they make a logical fraction of the complex problem. A problem that you can solve easily and quickly so now when you have broken down the problem for which can write a basic simple function. to write code in a TDD way you need to ask the WHAT question. What does this piece of code do? Then write a test for it. You test the output of the small piece of code. Another way to do this is by answering the SHOULD criteria. As in: this code SHOULD do WHAT. It SHOULD add 2 + 2. Does it do it?

Now TDD uses the 3 Factormethod of creating code - you write the test, watch it fail, then write AS LITTLE code as possible to make it pass, then refactor to make it aesthetic and move on. You see how you can see a tree diagram to break down the problem into smaller ones, and then write tests to all the breakdown specifications, problems or functionalities as you may prefer. This is the TDD way ... designing better code for tomorrow ... today.
